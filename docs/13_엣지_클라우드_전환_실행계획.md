# 13. 엣지–클라우드 연동 로드맵

이 문서는 **현재 구현과 확장 계획**을 한곳에 모은 참고용입니다. GCP 등 특정 벤더에 고정되지 않으며, 온프레미스 Windows 서버와 동일한 계약으로 이해하면 됩니다.

## 1) 역할 분담

### 엣지 (라즈베리파이, `edge/`)

- 카메라 캡처·전처리·YOLO 추론
- 즉시 판정·GPIO 알람(구성에 따라)
- 검사 결과·이미지 메타를 중앙 API로 전송
- (선택) 전송 실패 시 로컬 큐·재시도 — 구현 여부는 `edge` 소스의 전송기 모듈을 확인

### 클라우드/중앙 서버 (현재: `backend/` Spring Boot + MySQL)

- 검사 이력 영구 저장
- REST 조회·통계·이미지 경로 제공
- 향후: 사용자·권한, 다중 장비, MQTT 수신 등 확장

### 대시보드·키오스크

- `frontend/`: 운영자용 React 대시보드
- `frontend-kiosk/`: 현장 Blazor 터치 UI → 엣지 HTTP API 사용

## 2) API·스키마 계약 (원칙)

버전 접두어 예: **`/api/v1/inspections`** 와 기존 **`/api/inspections`** 병행 가능. 엣지 설정의 `SERVER_BASE_URL`·경로 접두어를 백엔드 배포와 맞출 것.

권장으로 문서화해 둔 핵심 필드 예시:

- `inspectionId` / 서버 발급 ID
- `deviceId`
- `timestamp` (`inspectedAt` 등)
- `verdict` 또는 `result` (PASS/FAIL)
- `defectType`, `confidence`
- `imagePath` 또는 Base64 동봉 후 서버가 파일 저장

실제 JSON 키 이름은 Spring DTO와 Python `InspectionPacket`(또는 동등 모델)을 **단일 소스**로 삼고, 이 문서는 개념만 유지합니다.

## 3) 통신 로드맵

| 단계 | 방식 | 비고 |
|------|------|------|
| 1 | HTTPS REST | 기본. 방화벽·TLS 인증서만 맞추면 됨 |
| 2 | MQTT 업/다운링크 | 실시간 제어·다중 구독이 필요해질 때 |

## 4) 운영 시 공통 체크

- 검사 이미지 저장 경로(`app.inspection-image-dir` 등)를 **디스크 용량·백업 정책**과 함께 설계
- 이미지 증가에 대비한 보관 주기·압축·썸네일
- 장비 인증: API 키·클라이언트 인증서·JWT 등 단계적 도입

## 5) .NET / GCP 등 이후 스택

ASP.NET Core·Blazor 단일 스택이나 GCP Cloud Run 배포는 **별도 브랜치·저장소에서 도입**할 수 있습니다. 이 저장소 루트의 [README.md](../README.md)에 안내된 것처럼, 해당 폴더가 없을 수 있으므로 실행 절차는 그 프로젝트의 README를 따릅니다.
